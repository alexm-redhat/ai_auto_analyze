================================================================================
CROSS-FRAMEWORK PERFORMANCE COMPARISON
vLLM (TARGET) vs SGLang
================================================================================
Model           : nvidia/Kimi-K2.5-NVFP4 (DeepSeekV3 arch, ~1T params)
GPU             : NVIDIA B200 x8 (TP=8)
Execution       : Decode, BS=1, ISL=4, OSL=1024
Block type      : MoE Decoder Layer (Layers 1-60)

TARGET (vLLM)   : wall = 97,919 ns | main-stream compute = 101,853 ns
Baseline (SGL)  : wall = 90,144 ns | main-stream compute =  94,719 ns
Delta           : +7,775 ns (+8.6%) wall time | +7,134 ns main-stream compute

================================================================================
EXECUTIVE SUMMARY
================================================================================

vLLM is 7,775 ns (8.6%) SLOWER per MoE decoder block than SGLang.

The gap is driven by five root-cause categories. Three are negative for vLLM
(opportunities), one is positive (existing advantage), and one is minor noise:

  NEG-1  MoE dual-stream design & shared expert TP policy   +7,232 ns  (7.4%)
  NEG-2  vLLM-only MoE overhead kernels on critical path    +4,352 ns  (4.4%)
  NEG-3  Explicit Q concat+scale Triton kernel              +1,696 ns  (1.7%)
  NEG-4  Attention kernel micro-differences (net)             +350 ns  (0.4%)
  POS-1  AllReduce + RMSNorm fusion                         -6,496 ns  (6.6%)
  ---------------------------------------------------------------
  NET (main-stream compute delta)                           +7,134 ns
  Residual (overlap / pipelining efficiency)                  +641 ns
  ---------------------------------------------------------------
  TOTAL WALL-TIME DELTA                                     +7,775 ns

  (Percentages are relative to vLLM wall time of 97,919 ns.)

KEY FINDINGS:

1. The LARGEST gap (+7,232 ns) comes from vLLM placing the routed MoE
   pipeline (37.1 us, including 10.7 us of routing overhead) on the
   main-stream critical path. SGLang instead places the shared expert
   pipeline (29.9 us) on its main stream, hiding the routing-heavy
   routed MoE on an alternate stream. This is enabled by SGLang setting
   the shared expert to tp_size=1, making it large enough to serve as
   a credible main-stream workload.

2. vLLM's AllReduce+RMSNorm fusion is its single biggest ADVANTAGE
   (-6,496 ns), fusing two AllReduce+RMSNorm+residual sequences into
   single Lamport-synchronized kernels. SGLang cannot enable this
   (cuMemCreate probe failed on B200 configuration).

3. vLLM has three overhead kernels on the critical path (+4,352 ns for
   workspace zero-fill and routing bias copy, plus +1,696 ns for Q
   concat+scale) that SGLang avoids entirely.

4. If NEG-1 and NEG-2 were addressed (adopting SGL's shared expert TP
   policy and eliminating overhead kernels), vLLM would be ~4.7 us
   FASTER than SGLang per block (thanks to allreduce fusion), rather
   than 7.8 us slower.

DETAILED PERFORMANCE DIFFERENCES
================================================================================
Sorted: negatives from worst to least-worst, then positives from most to least.
All deltas are from vLLM's perspective (positive = vLLM is slower).


================================================================================
NEG-1: MoE DUAL-STREAM DESIGN & SHARED EXPERT TP POLICY
================================================================================
Impact  : +7,232 ns on critical path (7.4% of vLLM wall time)
Category: Structural / design decision
Affects : MoE phase main-stream critical path

-------------------------------------------------------------------------------
WHAT IS HAPPENING
-------------------------------------------------------------------------------

Both frameworks execute the shared expert and routed MoE expert pipelines in
parallel on two CUDA streams. The key difference is WHICH pipeline is on the
main stream (critical path) and WHICH is hidden on the alternate stream:

  vLLM:  Routed MoE on MAIN stream (23)   = 37,119 ns
         Shared expert on secondary (3649) = 22,400 ns (fully hidden)
         MoE critical path = 37,119 ns + 4,352 ns overhead = 41,471 ns

  SGL:   Shared expert on MAIN stream (7)  = 29,887 ns
         Routed MoE on ALT stream (3488)   = 32,928 ns (mostly hidden)
         MoE critical path = max(29,887, 32,928) + sync + add = ~34,432 ns

The delta on the main-stream critical path: 37,119 - 29,887 = 7,232 ns.

The routed MoE pipeline is longer because it includes the routing overhead:
  - Router gate GEMM:            2,880 ns
  - Routing indices (topk-8):    7,808 ns
  - Subtotal routing:           10,688 ns  (no equivalent on shared expert path)

This routing overhead runs on vLLM's critical path but is HIDDEN on SGLang's
alternate stream.

-------------------------------------------------------------------------------
ROOT CAUSE: SHARED EXPERT TP POLICY
-------------------------------------------------------------------------------

The reason SGLang CAN put the shared expert on main stream (as a viable
critical-path workload that hides the routed MoE) is that SGLang does NOT
TP-shard the shared expert, making its GEMMs 8x larger per GPU:

  vLLM shared expert (TP-sharded):    gate_up [7168]->[512/rank],  down [256]->[7168]
  SGL shared expert (tp_size=1):      gate_up [7168]->[4096],      down [2048]->[7168]

vLLM's TP-sharded shared expert pipeline (22.4 us) is too short to be a
credible main-stream workload -- it finishes 14.7 us before the routed MoE,
wasting overlap potential.

SGLang's unsharded shared expert (29.9 us) is much closer in length to the
routed MoE (32.9 us), so both pipelines are well-balanced and overlap is
maximized.

================================================================================
NEG-2: vLLM-ONLY MoE OVERHEAD KERNELS ON CRITICAL PATH
================================================================================
Impact  : +4,352 ns on critical path (4.4% of vLLM wall time)
Category: Implementation overhead
Affects : MoE phase main-stream critical path

-------------------------------------------------------------------------------
WHAT IS HAPPENING
-------------------------------------------------------------------------------

vLLM launches two overhead kernels on its main stream before the MoE expert
computation that SGLang does not require:

  (a) MoE workspace zero-fill:  vectorized_elementwise_kernel  = 2,368 ns
      FillFunctor<unsigned char> zeros the MoE output workspace buffer.

  (b) MoE routing data copy:    vectorized_elementwise_kernel  = 1,984 ns
      bfloat16_copy_kernel copies the e_score_correction_bias tensor.

Both are on main stream 23 and directly add to vLLM's wall time.

-------------------------------------------------------------------------------
ROOT CAUSE (a): WORKSPACE ZERO-FILL
-------------------------------------------------------------------------------

vLLM delegates MoE workspace management to FlashInfer. The FlashInfer TRT-LLM
MoE kernel internally launches a zero-fill of the output accumulation buffer
before expert computation. This ensures clean accumulation across the top-8
expert outputs during the finalize step.

  File: vllm/model_executor/layers/fused_moe/experts/trtllm_nvfp4_moe.py, lines 159-168

    def workspace_shapes(self, ...):
        # The workspaces for this implementation are managed by flashinfer.
        workspace1 = (0,)
        workspace2 = (0,)
        output = (M, self.hidden_dim)
        return (workspace1, workspace2, output)

  The zero-fill kernel is launched internally by FlashInfer's
  trtllm_fp4_block_scale_routed_moe() before the routing + GEMM kernels.

SGLang avoids this overhead because its MoE runner allocates the output buffer
with torch.empty() (no zeroing) and relies on the FlashInfer kernel to fully
write all output elements:

  File: python/sglang/srt/layers/moe/moe_runner/flashinfer_trtllm.py, lines 631-634

    symm_output = torch.empty(              # NO explicit zeroing
        hidden_states.shape[0],
        hidden_states.shape[1],
        dtype=hidden_states.dtype,
        device=hidden_states.device,
    )

This difference suggests that vLLM's FlashInfer integration path (using the
modular kernel API) triggers internal workspace zeroing that SGLang's direct
wrapper avoids.

-------------------------------------------------------------------------------
ROOT CAUSE (b): ROUTING BIAS DATA COPY
-------------------------------------------------------------------------------

vLLM converts the e_score_correction_bias tensor to bfloat16 before passing it
to FlashInfer, which triggers a PyTorch copy kernel:

  File: vllm/model_executor/layers/fused_moe/experts/trtllm_nvfp4_moe.py, lines 334-335

    # Currently FI requires bfloat16 routing bias.
    # https://github.com/flashinfer-ai/flashinfer/issues/2909
    if e_score_correction_bias is not None:
        e_score_correction_bias = e_score_correction_bias.to(torch.bfloat16)

This .to(torch.bfloat16) conversion launches a bfloat16_copy_kernel (1,984 ns)
on the main stream. SGLang avoids this by handling the routing bias dtype
internally within the routingIndicesBlockKernel.

-------------------------------------------------------------------------------
OPPORTUNITY
-------------------------------------------------------------------------------

  - Workspace zero-fill: Align vLLM's FlashInfer integration with SGLang's
    approach (torch.empty output + kernel-guaranteed full writes), or upstream
    a FlashInfer fix to skip the zero-fill when the kernel writes all elements.
  - Routing bias copy: Pre-convert the bias to bfloat16 at model load time
    and cache it, eliminating the per-forward copy. Or upstream a FlashInfer
    fix (issue #2909) to accept non-bfloat16 routing bias.


================================================================================
NEG-3: EXPLICIT Q CONCAT+SCALE TRITON KERNEL
================================================================================
Impact  : +1,696 ns on critical path (1.7% of vLLM wall time)
Category: torch.compile artifact / kernel fusion gap
Affects : Attention phase main-stream critical path

-------------------------------------------------------------------------------
WHAT IS HAPPENING
-------------------------------------------------------------------------------

vLLM launches an explicit Triton kernel (torch.compile-generated) to
concatenate the absorbed query components (q_nope_out[512] + q_pe[64] -> 576
dims) and apply softmax scaling before the MLA decode attention kernel.

  Kernel: triton_poi_fused__to_copy_cat_clamp_mul_reciprocal...
  Stream: 23 (main), Duration: 1,696 ns

SGLang does NOT have this kernel. Instead, it passes q_nope_out and q_pe as
separate tensors to the TRT-LLM MLA attention kernel, which handles
concatenation and scaling internally.

-------------------------------------------------------------------------------
ROOT CAUSE
-------------------------------------------------------------------------------

vLLM's torch.compile path generates this kernel because the MLA attention code
explicitly concatenates Q before calling the attention backend:

  File: vllm/model_executor/layers/attention/mla_attention.py, lines 740-748

    if fp8_attention and self.impl.supports_quant_query_input:
        mqa_q = self._decode_concat_quant_fp8_op(
            mqa_ql_nope, mqa_q_pe, self._q_scale       # Concat + FP8 quant
        )
    else:
        mqa_q = (mqa_ql_nope, mqa_q_pe)                # Tuple (no concat)

  The _DecodeConcatQuantFP8 class (lines 1171-1197) implements concat+quant:

    def forward(self, decode_ql_nope, decode_q_pe, scale, ...):
        decode_q0 = torch.cat((decode_ql_nope, decode_q_pe), dim=-1)  # Concat
        decode_q_flat = decode_q0.reshape(decode_q0.shape[0], -1)
        decode_q, _ = quant_fn(self, decode_q_flat, scale, ...)       # Quant
        return decode_q.view(decode_q0.shape)

  torch.compile fuses torch.cat + reshape + quant + view into a single Triton
  pointwise kernel, but it still runs as a separate kernel launch.

SGLang avoids this by passing Q components directly to the TRT-LLM attention:

  File: python/sglang/srt/models/deepseek_common/.../forward_mla.py, lines 406-419

    attn_output = self.attn_mqa(
        q_nope_out,                  # Passed separately
        q_pe,                        # Passed separately
        kv_cache, ...
    )

  The TRT-LLM MLA decode kernel (fmhaSm100fKernel) natively accepts separate
  q_nope and q_pe components and handles concatenation+scaling internally.

-------------------------------------------------------------------------------
OPPORTUNITY
-------------------------------------------------------------------------------

Modify vLLM's MLA attention backend to accept q_nope_out and q_pe as separate
inputs (matching how the TRT-LLM kernel natively operates), eliminating the
need for the explicit concat+scale kernel.


================================================================================
NEG-4: ATTENTION KERNEL MICRO-DIFFERENCES
================================================================================
Impact  : +350 ns on critical path (0.4% of vLLM wall time)
Category: Kernel-level variance / minor implementation differences
Affects : Attention phase main-stream critical path

-------------------------------------------------------------------------------
WHAT IS HAPPENING
-------------------------------------------------------------------------------

Several attention-phase kernels show small duration differences between
frameworks, even when using identical kernel binaries. These are individually
small but collectively add +350 ns:

  Component       vLLM (ns)  SGL (ns)  Delta     Notes
  -------------   ---------  --------  ------    ---------------------
  Q-B proj GEMM     5,792     5,504     +288     Same cuBLAS nvjet kernel
  KV cache update   2,079     1,664     +415     Different kernels, diff work
  Q-A RMSNorm       4,191     3,968     +223     2 Triton vs 1 FlashInfer
  W_UV BMM          4,000     3,776     +224     Same cuBLAS nvjet kernel
  MLA attention     9,536     9,376     +160     Same FMHA kernel family
  QKV-A proj        7,584     7,520      +64     Same PDL kernel
  kv_a+RoPE         2,112     2,176      -64     Different fusion patterns
  O proj            5,568     5,792     -224     Same cuBLAS nvjet kernel
  W_UK BMM          2,880     3,616     -736     Same kernel, diff layout
  -------------   ---------  --------  ------
  Net                                   +350

Notable sub-items:

(a) W_UK/W_kc BMM layout optimization (vLLM faster by 736 ns):
    vLLM uses NNT layout: torch.bmm(q_nope, W_UK_T) -- no runtime transpose.
    SGLang uses TNT layout: torch.bmm(q_nope.transpose(0,1), w_kc) -- runtime
    transpose on Q creates a non-contiguous input, hurting cache efficiency.

    vLLM pre-transposes the weight at initialization:
      File: mla_attention.py, line 903
        self.W_UK_T = W_UK.permute(1, 2, 0)   # (L,N,P) -> (N,P,L)

    SGLang transposes Q at each forward pass:
      File: forward_mla.py, line 311
        q_nope_out = torch.bmm(q_nope.transpose(0, 1), self.w_kc)

(b) Q-A RMSNorm (vLLM slower by 223 ns):
    vLLM: torch.compile splits RMSNorm into 2 Triton kernels (pointwise 1,760
    + reduction 2,431 = 4,191 ns total) because Inductor cannot fuse the
    reduction with the pointwise in a single kernel.
    SGLang: Single FlashInfer CUTLASS RMSNorm kernel (3,968 ns).

(c) KV cache update (vLLM slower by 415 ns):
    vLLM: concat_and_cache_mla_kernel (2,079 ns) performs concat + FP8
    quantization + paged cache write in one kernel.
    SGLang: set_mla_kv_buffer_kernel (1,664 ns) only writes to pages; FP8
    quantization was already done in the prior RopeQuantizeKernel.
    SGLang's fusion of RoPE+FP8_quant into one kernel allows the cache write
    to be simpler and faster.

(d) Identical kernels with small variance (Q-B proj +288, W_UV BMM +224,
    MLA attention +160, QKV-A +64, O proj -224):
    These use the same cuBLAS nvjet or FMHA kernel binary. The differences
    are likely caused by GPU SM contention from preceding/concurrent operations,
    L2 cache state, and CUDA graph replay timing variation. No actionable
    root cause.

-------------------------------------------------------------------------------
OPPORTUNITY
-------------------------------------------------------------------------------

  - W_UK BMM: SGLang could adopt vLLM's NNT layout (pre-transposed weight) to
    save 736 ns. This is a positive differentiator for vLLM.
  - Q-A RMSNorm: vLLM could use a standalone FlashInfer/CUTLASS RMSNorm kernel
    instead of relying on torch.compile to split it into 2 kernels. Potential
    saving: ~223 ns.
  - KV cache: vLLM could adopt SGLang's pattern of fusing FP8 quant into the
    RoPE kernel, simplifying the cache write kernel. Potential saving: ~415 ns.


================================================================================
POS-1: ALLREDUCE + RMSNORM FUSION
================================================================================
Impact  : -6,496 ns on critical path (6.6% of vLLM wall time -- ADVANTAGE)
Category: Kernel fusion / communication optimization
Affects : AllReduce + RMSNorm operations at layer boundaries

-------------------------------------------------------------------------------
WHAT IS HAPPENING
-------------------------------------------------------------------------------

vLLM fuses AllReduce + RMSNorm + residual-add into a single kernel
(allreduce_fusion_kernel_oneshot_lamport) at two points per block:

  (1) Input layernorm:  AllReduce of prev-layer MoE output + RMSNorm + residual
  (2) Post-attn layernorm: AllReduce of o_proj output + RMSNorm + residual

SGLang runs these as SEPARATE operations because its allreduce fusion is
disabled (cuMemCreate probe failed on this B200 configuration):

  (1a) Previous block's MoE AllReduce:  cross_device_reduce_1stage = 6,656 ns
  (1b) Input RMSNorm + residual:        fused_add_rmsnorm           = 3,328 ns
  (2a) Attention AllReduce:             cross_device_reduce_1stage = 8,256 ns
  (2b) Post-attn RMSNorm + residual:   fused_add_rmsnorm           = 3,200 ns

  Savings breakdown:
    Input norm:     SGL (6,656 + 3,328 = 9,984) - vLLM (7,040) = 2,944 ns saved
    Post-attn norm: SGL (8,256 + 3,200 = 11,456) - vLLM (7,904) = 3,552 ns saved
    Total:                                                        = 6,496 ns saved

The fused kernel eliminates inter-kernel launch overhead, extra global memory
round-trips, and synchronization barriers between allreduce and norm operations.

-------------------------------------------------------------------------------
ROOT CAUSE
-------------------------------------------------------------------------------

vLLM uses a torch.compile fusion pass that pattern-matches AllReduce followed
by RMSNorm and replaces both with a single FlashInfer fused kernel:

  File: vllm/compilation/passes/fusion/allreduce_rms_fusion.py

  Pattern (lines 361-366, AllReduceFusedAddRMSNormPattern):

    def pattern(residual, input, weight):
        allreduce_output = tensor_model_parallel_all_reduce(input)
        rms, residual = self.rmsnorm_matcher(allreduce_output, weight, residual)
        return rms, residual

  Replacement (lines 368-385):

    def replacement(residual, input, weight):
        allreduce = auto_functionalized(
            flashinfer_trtllm_fused_allreduce_norm,
            allreduce_in=input,
            residual=residual,
            rms_gamma=weight,
            rms_eps=self.epsilon,
            pattern_code=AllReduceFusionPattern.kARResidualRMSNorm,
            ...
        )
        return allreduce[1], allreduce[2]

  The fused kernel uses Lamport one-shot synchronization for low-latency
  8-GPU allreduce (lines 136-145):

    use_oneshot = (
        max_one_shot_size is None
        or current_tensor_size <= max_one_shot_size * MiB
    )

  For SM100 (Blackwell) with world_size=8, one-shot is enabled for tensors
  up to 1 MB (line 91), which covers the 7168-dim hidden states (28 KB).

  Enable condition (vllm/config/vllm.py):

    def enable_allreduce_rms_fusion(cfg):
        return (
            cfg.parallel_config.tensor_parallel_size > 1
            and current_platform.is_cuda()
            and has_flashinfer()
            and (
                current_platform.is_device_capability_family(100)  # Blackwell
                or current_platform.is_device_capability(90)       # Hopper
            )
            and cfg.parallel_config.data_parallel_size == 1
        )

SGLang has a similar allreduce_fusion code path (communicator.py) but it was
disabled at runtime because the cuMemCreate probe (required for symmetric
memory allocation by the FlashInfer allreduce kernel) failed on this B200
configuration. As a result, SGLang falls back to separate custom allreduce
(cross_device_reduce_1stage) + FlashInfer fused_add_rmsnorm.

-------------------------------------------------------------------------------
SIGNIFICANCE
-------------------------------------------------------------------------------

This is vLLM's single largest per-block advantage. The 6,496 ns saving fully
offsets NEG-2 (+4,352 ns) and NEG-3 (+1,696 ns) combined. If SGLang could
enable its own allreduce fusion, the wall-time gap would widen further.

================================================================================
ACTIONABLE RECOMMENDATIONS FOR vLLM (TARGET)
================================================================================

Sorted by expected impact (highest first):

1. ADOPT UNSHARDED SHARED EXPERT POLICY (addresses NEG-1)
   Expected saving: ~7,200 ns per block
   - Set shared expert tp_size=1 (matching SGLang's approach) when using
     FlashInfer TRTLLM MoE backend
   - Swap stream assignment: shared expert -> main stream, routed MoE -> alt
   - This moves the 10.7 us routing overhead off the critical path
   - Reference: SGLang deepseek_v2.py lines 486-512 (tp_size=1 condition)
     and lines 626-666 (forward_normal_dual_stream)

2. ELIMINATE MoE OVERHEAD KERNELS (addresses NEG-2)
   Expected saving: ~4,350 ns per block
   - Workspace zero-fill: Modify FlashInfer integration to skip the output
     buffer zeroing (the finalize kernel fully writes all elements)
   - Routing bias copy: Pre-convert e_score_correction_bias to bfloat16 at
     model initialization and cache it, avoiding per-forward .to() copy
   - Reference: FlashInfer issue #2909 for routing bias dtype

3. FUSE Q CONCAT+SCALE INTO ATTENTION KERNEL (addresses NEG-3)
   Expected saving: ~1,700 ns per block
   - Pass q_nope_out and q_pe as separate tensors to the MLA decode backend
   - The TRT-LLM fmhaSm100f kernel natively supports separate Q components
   - This eliminates the torch.compile-generated Triton concat kernel

4. MINOR ATTENTION OPTIMIZATIONS (addresses NEG-4 partially)
   Expected saving: ~640 ns per block
   - Use standalone FlashInfer RMSNorm for Q-A norm instead of Triton split
   - Fuse RoPE + FP8 quantization into a single kernel (like SGLang's
     RopeQuantizeKernel) to simplify KV cache write

5. PROTECT ALLREDUCE FUSION ADVANTAGE (preserves POS-1)
   - The 6,496 ns allreduce fusion saving is the critical advantage; ensure
     it remains enabled for all supported configurations
   - If items 1-4 are implemented, vLLM would be ~4.7 us faster per block
