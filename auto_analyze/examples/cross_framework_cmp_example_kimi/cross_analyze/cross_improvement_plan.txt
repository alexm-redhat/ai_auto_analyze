================================================================================
IMP-1: UNSHARDED SHARED EXPERT + STREAM SWAP
================================================================================
Impact    : -7,232 ns (-7.4% of block time)
Complexity: Medium-High (structural change to MoE layer)
Priority  : HIGHEST — recovers 93% of the gap

--------------------------------------------------------------------------------
ROOT CAUSE
--------------------------------------------------------------------------------

vLLM TP-shards the shared expert, producing a very short pipeline (22.4 us)
that runs on an auxiliary stream. The routed MoE pipeline (37.1 us, including
10.7 us of routing overhead) runs on the main stream, making it the critical
path. SGLang disables TP for the shared expert (tp_size=1), making it 29.9 us
— long enough to be the main-stream workload — and hides the routed MoE
(32.9 us) on an alternate stream.

  vLLM current:  main = routed MoE (37.1 us) | aux = shared expert (22.4 us)
  SGLang:        main = shared expert (29.9 us) | alt = routed MoE (32.9 us)
  Target:        main = shared expert (~29.9 us) | alt = routed MoE (~35 us)

    def _forward_impl_swapped_streams(self, layer, hidden_states,
                                       router_logits, shared_experts_input):
        """Dual-stream execution with shared expert on MAIN stream and
        routed MoE on AUX stream. Used when shared expert is unsharded
        (disable_tp=True) and provides better overlap.

        Reference: SGLang deepseek_v2.py forward_normal_dual_stream()
        """
        aux = self._shared_experts._stream
        assert aux is not None

        # Sync aux stream start point
        aux.wait_stream(current_stream())

        # MAIN STREAM: Run shared expert
        shared_output = self._shared_experts._layer(shared_experts_input)

        # AUX STREAM: Run gate + routing + fused MoE experts
        with torch.cuda.stream(aux):
            if self.gate is not None:
                router_logits, _ = self.gate(hidden_states)

            if self.quant_method.is_monolithic:
                fused_out = self.quant_method.apply_monolithic(
                    layer=layer, x=hidden_states, router_logits=router_logits,
                )
            else:
                topk_weights, topk_ids = self.router.select_experts(
                    hidden_states=hidden_states, router_logits=router_logits,
                )
                fused_out = self.quant_method.apply(
                    layer=layer, x=hidden_states,
                    topk_weights=topk_weights, topk_ids=topk_ids,
                )

        # SYNC: wait for routed MoE to finish
        current_stream().wait_stream(aux)

        return shared_output, fused_out

    def _should_use_swapped_dual_stream(self, shared_experts_input):
        """Use swapped streams when shared expert is unsharded."""
        if self._shared_experts is None:
            return False
        if self._shared_experts._stream is None:
            return False
        if shared_experts_input is None:
            return False
        # Check if shared expert is unsharded (disable_tp was True)
        layer = self._shared_experts._layer
        if hasattr(layer, 'gate_up_proj') and hasattr(layer.gate_up_proj, 'tp_size'):
            return layer.gate_up_proj.tp_size == 1
        return False

================================================================================
IMP-2b: ELIMINATE MOE WORKSPACE ZERO-FILL
================================================================================
Impact    : -2,368 ns (-2.4% of block time)
Complexity: Medium (requires understanding FlashInfer internals)
Priority  : HIGH — significant impact, medium effort

--------------------------------------------------------------------------------
ROOT CAUSE
--------------------------------------------------------------------------------

vLLM's MoE pipeline launches a vectorized_elementwise_kernel (FillFunctor
<unsigned char>, 2,368 ns) to zero the MoE output workspace buffer before
expert computation. This happens internally within FlashInfer's
trtllm_fp4_block_scale_routed_moe() / trtllm_fp4_block_scale_moe() call.

SGLang avoids this by allocating the output buffer externally with
torch.empty() (no zeroing) and passing it to the FlashInfer kernel.

  SGLang (flashinfer_trtllm.py lines 631-634):
      symm_output = torch.empty(
          hidden_states.shape[0], hidden_states.shape[1],
          dtype=hidden_states.dtype, device=hidden_states.device,
      )

The FlashInfer MoE kernel fully writes all output elements via the finalize
kernel (weighted combination of top-8 expert outputs), so pre-zeroing is
unnecessary.

--------------------------------------------------------------------------------
STEP-BY-STEP IMPLEMENTATION
--------------------------------------------------------------------------------

STEP 1: Identify the FlashInfer API difference
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

vLLM uses the modular kernel API path that triggers internal workspace zeroing:
  File: vllm/model_executor/layers/fused_moe/experts/trtllm_nvfp4_moe.py

  Lines 209-240 (modular apply):
      flashinfer.fused_moe.trtllm_fp4_block_scale_routed_moe(
          topk_ids=packed_tensor,
          ...

      )

  Lines 340-370 (monolithic apply):
      flashinfer.fused_moe.trtllm_fp4_block_scale_moe(

The zero-fill is triggered by FlashInfer's internal C++ code before the
expert computation begins. The kernel creates an internal workspace buffer

================================================================================
IMP-3: OPTIMIZE Q CONCAT+SCALE (REPLACE TRITON KERNEL)
================================================================================
Impact    : -1,696 ns (-1.7% of block time)
Complexity: Medium
Priority  : MEDIUM — meaningful impact, targeted change

--------------------------------------------------------------------------------
ROOT CAUSE
--------------------------------------------------------------------------------

vLLM's torch.compile generates an explicit Triton pointwise kernel to
concatenate q_nope_out[512] + q_pe[64] -> q[576] and apply softmax scaling.
This kernel (triton_poi_fused__to_copy_cat_clamp_mul_reciprocal, 1,696 ns)
runs on the main stream before the MLA decode attention kernel.

SGLang avoids this overhead by:
(a) Passing separate q_nope_out and q_pe to the attention backend, which
    handles concatenation internally using an optimized CUDA kernel.
(b) Using concat_mla_absorb_q — a warp-vectorized CUDA kernel optimized
    for the specific 512+64 -> 576 shape (from sgl-kernel).

The FlashInfer trtllm_batch_decode_with_kv_cache_mla kernel requires a
pre-concatenated query tensor, so the concat must happen — the question
is HOW efficiently.

  SGLang (forward_mla.py line 406-419 + trtllm_mla_backend.py lines 803-809):
      # Passes separate q_nope, q_pe to backend
      attn_output = self.attn_mqa(q_nope_out, k_nope, k_nope,
                                   forward_batch, q_rope=q_pe, k_rope=k_pe)
      # Backend uses optimized fused concat kernel:
      query = concat_mla_absorb_q_general(q_nope, q_rope_reshaped)

  SGLang (utils.py lines 470-474):
      def concat_mla_absorb_q_general(q_nope, q_rope):
          if _is_cuda and q_nope.shape[-1] == 512 and q_rope.shape[-1] == 64:
              return concat_mla_absorb_q(q_nope, q_rope)  # Fused CUDA kernel
          else:
              return torch.cat([q_nope, q_rope], dim=-1)

  The fused CUDA kernel uses warp-level int4 vectorized loads/stores,
  processing one (batch, head) row per warp with ~2x throughput vs torch.cat.

STEP 1: Add an optimized concat kernel to vLLM
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Port SGLang's concat_mla_absorb_q CUDA kernel to vLLM, or use the
FlashInfer equivalent if available.

Option A: Register a custom op that wraps an optimized concat:

    File: vllm/model_executor/layers/attention/mla_attention.py (new utility)

    import torch

    @torch.library.custom_op("vllm::concat_mla_q", mutates_args=())
    def concat_mla_q(
        q_nope: torch.Tensor,  # [..., 512]
        q_pe: torch.Tensor,    # [..., 64]
    ) -> torch.Tensor:
        """Optimized concat for MLA query components.
        Registered as custom_op to prevent torch.compile from fusing
        it into a slower Triton kernel."""
        return torch.cat([q_nope, q_pe], dim=-1)

    @concat_mla_q.register_fake
    def _(q_nope, q_pe):
        out_dim = q_nope.shape[-1] + q_pe.shape[-1]
        return torch.empty(
            *q_nope.shape[:-1], out_dim,
            dtype=q_nope.dtype, device=q_nope.device,
        )

By registering as a custom_op, torch.compile treats it as an opaque
operation and does NOT try to decompose/fuse it into a Triton kernel.
The torch.cat inside runs as a single efficient PyTorch cat kernel
(~500 ns) instead of the generated Triton kernel (~1,696 ns).

Option B (higher performance): Port the CUDA kernel from SGLang:

    Port sgl-kernel's concat_mla.cuh kernel to vllm. The kernel uses
    warp-level int4 vectorized loads for the 512-dim tensor and int
    loads for the 64-dim tensor, storing to output with L1 bypass hints.
