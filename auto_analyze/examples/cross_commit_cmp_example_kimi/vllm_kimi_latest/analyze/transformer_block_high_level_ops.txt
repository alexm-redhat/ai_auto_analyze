================================================================================
NOTES ON FUSIONS AND CROSS-LAYER EFFECTS
================================================================================

1. ALLREDUCE + RMS NORM FUSION (allreduce_rms):
   This is the most impactful fusion. It replaces 2-3 separate kernels with a
   single flashinfer_trtllm_fused_allreduce_norm kernel. There are two instances
   per transformer block:
     (a) After o_proj: allreduce + residual_add + post_attention_layernorm
     (b) After MLP/MoE: allreduce + residual_add + next_layer.input_layernorm
   Instance (b) is a CROSS-LAYER fusion: the allreduce from the current layer's
   MLP output is fused with the NEXT layer's input layernorm. The last layer's
   instance (b) fuses with the final model norm (DeepseekV2Model.norm).

   Supported patterns (from allreduce_rms_fusion.py):
     - AllReduceRMSNormPattern (allreduce + rms_norm)
     - AllReduceFusedAddRMSNormPattern (allreduce + residual_add + rms_norm)
     - AllReduceFusedRMSNormStaticQuantFP8Pattern (allreduce + rms_norm + FP8 quant)
     - AllReduceFusedAddRMSNormStaticQuantFP8Pattern (all three + residual)
     - FP4 variants for SM100+ (kARResidualRMSNormFP4Quant)

2. ACTIVATION + QUANTIZATION FUSION (act_quant):
   Fuses SiLU-and-Mul with subsequent FP4 or FP8 quantization. Applies to the
   dense MLP layers (layer 0) and shared experts in MoE layers when the
   activation output feeds into an NVFP4-quantized down_proj. Does NOT apply
   inside the TRTLLM MoE kernel (which handles activation internally).

   Patterns (from act_quant_fusion.py):
     - SiluMulFp8StaticQuantPattern (silu_and_mul + FP8 static quant)
     - SiluMulNvfp4QuantPattern (silu_and_mul + NVFP4 quant)
     - SiluMulBlockQuantPattern (silu_and_mul + FP8 dynamic block quant)

3. SHARED EXPERT STREAM OVERLAP:
   The shared expert in MoE layers can execute on a separate CUDA stream
   (MULTI_STREAM_OVERLAPPED mode), overlapping with the gate + TRTLLM monolithic
   expert kernel on the main stream. The sync point is after the TRTLLM kernel
   completes, where the main stream waits for the aux stream before adding
   shared + routed outputs.

4. MIN-LATENCY GEMM FOR FUSED QKV-A:
   The fused_qkv_a_proj uses a specialized dsv3_fused_a_gemm kernel for <= 16
   tokens (typical decode batch). This kernel uses PDL (Programmatic Dependent
   Launch) for reduced launch overhead. For larger batches (prefill), it falls
   back to standard cuBLAS BF16 GEMM via F.linear.

5. NVFP4 LINEAR KERNEL FLOW (FlashInferCutlassNvFp4LinearKernel):
   For NVFP4-quantized non-MoE linear layers (e.g., q_b_proj, o_proj):
     (a) Dynamic input quantization: scaled_fp4_quant(x, scale) -> FP4 packed + block scales
     (b) FP4 GEMM: cutlass_scaled_fp4_mm(x_fp4, weight, x_scales, w_scales, alpha)
     (c) Output in BF16, sliced to actual output size
   The act_quant fusion can eliminate step (a) by fusing it with the preceding
   activation function.

6. TRTLLM FP4 MoE KERNEL INTERNALS (trtllm_fp4_block_scale_moe):
   A single monolithic GPU kernel that fuses:
     (a) Router: softmax on logits, noaux_tc top-8 selection with bias correction
     (b) Dispatch: route tokens to local 48 experts (out of 384 global)
     (c) GEMM1 (FP4): gate+up projection [7168 -> 4096] with block-scale dequant
     (d) Activation: gated SiLU (silu(gate) * up)
     (e) GEMM2 (FP4): down projection [2048 -> 7168] with block-scale dequant
     (f) Combine: weighted sum of expert outputs by routing weights
   Scales are pre-fused: g1_scale_c = w13_weight_scale_2 * w13_input_scale,
   g2_alphas = w2_weight_scale_2, etc. Routing factor (2.827) applied internally.

7. KV CACHE FORMAT:
   MLA stores compressed KV vectors (kv_lora_rank + qk_rope_head_dim = 576 dims)
   in FP8_E4M3 format in the KV cache. Block size is 32 tokens. This is much
   more memory-efficient than standard MHA which would store full K and V heads.

8. DECODE MLA PATH (MQA):
   Instead of expanding compressed KV via kv_b_proj (as in prefill), the decode
   path compresses Q into the latent space:
     - Q_nope @ W_UK_T: projects Q to latent KV space (kv_lora_rank=512)
     - MQA attention operates in compressed space (576 dims)
     - V_up_proj: projects attention output back to value space (v_head_dim=128)
   This is more memory-efficient for decode (bandwidth-bound), while prefill
   uses expanded KV for better compute utilization.
