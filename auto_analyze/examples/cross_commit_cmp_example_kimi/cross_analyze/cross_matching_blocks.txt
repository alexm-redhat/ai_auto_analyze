The two traces use different block boundary conventions due to changes in how vLLM's
torch.compile graph splitting and CUDA graph capture evolved between versions:

  Trace [0] (v0.20.2) block scope:
    STARTS at: fused_qkv_a_proj_gemm (first attention op)
    ENDS at:   fused_allreduce_rms_post_moe (cross-layer: MoE allreduce + NEXT layer's input RMSNorm)
    Interpretation: The block encompasses the FULL logical transformer layer, because the
    input RMSNorm is fused into the PREVIOUS layer's trailing allreduce (last op of prev block).

  Trace [1] (v0.16.0) block scope:
    STARTS at: moe_scale_add_shared (PREVIOUS layer's MoE combine step)
              + fused_allreduce_input_rmsnorm (allreduce + THIS layer's input RMSNorm)
    ENDS at:   moe_finalize_combine (MoE finalize, but WITHOUT the trailing allreduce)
    Interpretation: The block is shifted — it includes 2 ops from the previous layer at the
    start and excludes 2 equivalent ops (add+allreduce) at the end.

  NET EFFECT: Both blocks represent exactly ONE logical transformer layer's worth of compute.
  The operations are the SAME set, just with different cut points in the circular chain.

  KEY FINDINGS — MoE ROUTING:
  1. Gate Router GEMM (C1): v0.20.2 uses dsv3_router_gemm, a specialized min-latency CUDA kernel
     adapted from NVIDIA TRT-LLM for DeepSeek V3 router GEMM at small batch sizes (<=16 tokens).
     Uses optimized grid=(384,1,1) vs v0.16.0's standard nvjet at grid=(48,1,1).
     Savings: 7,967 ns (73.2%).
  2. Routing Consolidation (C5 vs C5a-d): v0.20.2 consolidates 4 separate routing kernels
     (sigmoid + bias_correction + topk_routing + indices_clustering) into a single monolithic
     routingIndicesBlockKernel from the TRTLLM MoE library. This eliminates 3 kernel launch
     overheads and enables data-local computation.
     Savings: 7,648 ns (50.1%).
  3. Two new small ops (C2, C4) were added for TRTLLM MoE monolithic kernel preparation,
     costing 3,296 ns combined — partially offsetting the routing consolidation savings.

4.3  STRUCTURAL CHANGES (1:N or N:1 mappings)
----------------------------------------------------------------------------------------------------------------------------

  CHANGE 1: MoE Routing Consolidation (4 kernels → 1 kernel)

    v0.16.0 (4 separate kernels, total 15,264 ns on main stream):
      moe_routing_sigmoid          (2,400 ns)  — sigmoid activation on router logits
      moe_routing_bias_correction  (3,200 ns)  — add e_score_correction_bias
      moe_topk_routing_main        (4,352 ns)  — top-8 expert selection
      moe_routing_indices          (5,312 ns)  — token-to-expert index clustering

    v0.20.2 (1 monolithic kernel, 7,616 ns on main stream):
      moe_routing_dispatch         (7,616 ns)  — TRTLLM routingIndicesBlockKernel
                                                  Fuses: sigmoid + bias + topk + indices + dispatch

    Savings: 15,264 - 7,616 = 7,648 ns (50.1%)
    Root cause: TRTLLM MoE monolithic kernel (MoEPrepareAndFinalizeNoDPEPMonolithic) internalizes
    routing computation, eliminating 3 kernel launch overheads and enabling data-local processing.

  CHANGE 2: QKV-A Projection Kernel Replacement

    v0.16.0: nvjet_tst_16x64_64x16_4x1_v_bz_TNN (generic CUTLASS BF16 GEMM, 18,112 ns)
    v0.20.2: dsv3_fused_a_gemm (specialized PDL min-latency GEMM, 7,328 ns)

    Savings: 10,784 ns (59.5%)
    Root cause: New DeepSeekV2FusedQkvAProjLinear class uses dsv3_fused_a_gemm kernel (from
    csrc/dsv3_fused_a_gemm.cu) with Programmatic Dependent Launch for reduced kernel launch
    latency at small batch sizes (<=16 tokens). Falls back to cuBLAS for larger batches.

  CHANGE 3: Gate Router GEMM Kernel Replacement

    v0.16.0: nvjet_tst_8x64_64x16_4x1_v_bz_TNN (generic CUTLASS BF16 GEMM, grid 48x1x1, 10,880 ns)
    v0.20.2: dsv3_router_gemm (specialized router GEMM, grid 384x1x1, 2,913 ns)

    Savings: 7,967 ns (73.2%)
    Root cause: New GateLinear class (vllm/model_executor/layers/fused_moe/router/gate_linear.py)
    dispatches to dsv3_router_gemm (from csrc/moe/dsv3_router_gemm_float_out.cu) for SM90+ GPUs
    at batch<=16 with specific dim combinations (7168x384). This kernel is optimized from NVIDIA's
    TRT-LLM DeepSeek V3 RouterGemm implementation.

  CHANGE 4: Allreduce Memcpy Elimination

    v0.16.0: memcpy32_post kernel (1,056 ns) after every fused_allreduce for Lamport signaling
    v0.20.2: Not present — Lamport protocol updated to not require separate memcpy

    Savings: 1,056 ns per allreduce (2,112 ns per layer with 2 allreduces)

SECTION 7: SUMMARY
================================================================================================================================

  Wall-time improvement: 26,240 ns (21.3% faster in v0.20.2)

  The performance gain from v0.16.0 → v0.20.2 is dominated by THREE kernel replacements:
    1. dsv3_fused_a_gemm for QKV-A projection     (-10.8 us, 41% of total savings)
    2. dsv3_router_gemm for MoE gate routing       ( -8.0 us, 30% of total savings)
    3. Monolithic TRTLLM routing dispatch           ( -7.6 us, 29% of total savings)

  These are all latency-optimized kernels specifically designed for DeepSeek V3/Kimi K2.5
  decode at small batch sizes (BS<=16). They replace generic CUTLASS/nvjet BF16 GEMMs and
  separate routing kernels with fused, low-overhead implementations.

  Additional contributions:
    - Shared expert down GEMM CUTLASS tuning        (-4.8 us)
    - Allreduce Lamport + memcpy improvements        (-2.6 us)
    - Minor regressions in standard nvjet GEMMs      (+1.8 us, partially offsetting)
    - New buffer init/cast ops for TRTLLM format     (+6.0 us, partially offsetting)

  Operation count: Both blocks have 29 GPU operations, but the composition differs:
    - v0.20.2 added 4 new ops (buffer inits + BF16 cast) = +6.0 us
    - v0.20.2 removed 5 ops (memcpy + 4 routing kernels) = -16.3 us
    - Net structural change: -10.3 us from kernel fusion/elimination
