 A.1  QKV-A Projection GEMM  [Match B1]
      TARGET: 7,328 ns  |  Baseline: 18,112 ns  |  Delta: -10,784 ns (-59.5%)
      Category: MAJOR IMPROVEMENT — largest single-op saving
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  The fused QKV low-rank projection GEMM [7168 -> 2112] was replaced from a generic cuBLAS/
  nvjet CUTLASS BF16 GEMM to a purpose-built, warp-specialized, min-latency CUDA kernel.

  Baseline (v0.16.0):
    Kernel: nvjet_tst_16x64_64x16_4x1_v_bz_TNN  (generic cuBLASLt CUTLASS BF16 GEMM)
    Grid: (132,1,1)  |  Duration: 18,112 ns
    Dispatch path: MergedColumnParallelLinear -> F.linear -> cuBLAS

  TARGET (v0.20.2):
    Kernel: dsv3_fused_a_gemm  (specialized PDL min-latency GEMM from TRT-LLM)
    Grid: (132,1,1)  |  Duration: 7,328 ns
    Dispatch path: DeepSeekV2FusedQkvAProjLinear -> ops.dsv3_fused_a_gemm

  ROOT CAUSE:
  The new kernel (csrc/dsv3_fused_a_gemm.cu) is a hand-tuned CUDA kernel adapted from
  NVIDIA TRT-LLM's DeepSeek V3 minimum-latency kernels, via SGLang's sgl-kernel. It is
  specifically designed for the exact problem shape [M<=16, K=7168, N=2112] at decode.

  Five key architectural features make it 2.5x faster than generic cuBLAS:

  (1) WARP SPECIALIZATION (4 load warps + 4 compute warps):
      The kernel uses 256 threads (8 warps) split into two groups: warps 0-3 are dedicated
      data loaders (global -> shared memory), and warps 4-7 are dedicated MMA compute warps.
      They communicate via hardware mbarrier barriers. This eliminates the load/compute
      serialization present in generic CUTLASS cooperative-style kernels.

      Source: csrc/dsv3_fused_a_gemm.cu:
      ```cpp
      if (warp_idx < 2) {
          a_loader.issue_mainloop();      // Warps 0-1: load matrix A
      } else if (warp_idx < 4) {
          b_loader.issue_mainloop();      // Warps 2-3: load matrix B
      } else {
          mma_computer.issue_mainloop();  // Warps 4-7: Tensor Core MMA
          mma_computer.epi();             // Epilogue: reduce + write output
      }
      ```

  (2) COMPILE-TIME SHAPE SPECIALIZATION:
      The kernel is hardcoded for kHdIn=7168, kHdOut=2112. All loop bounds, shared memory
      sizes, and register allocations are determined at compile time. cuBLAS must select tile
      configurations via runtime heuristics, adding overhead and often choosing suboptimal tiles.

      Source: csrc/dsv3_fused_a_gemm.cu:
      ```cpp
      TORCH_CHECK(num_tokens >= 1 && num_tokens <= 16);
      TORCH_CHECK(hd_in == kHdIn);    // == 7168
      TORCH_CHECK(hd_out == kHdOut);  // == 2112
      ```

  (3) DUAL TILE-N VARIANTS for optimal small-batch coverage:
      Two kernel variants are compiled: tile_n=8 for 1-8 tokens, tile_n=16 for 9-16 tokens.
      cuBLAS uses the same (larger) tile for all batch sizes, wasting resources at BS=1.

      Source: csrc/dsv3_fused_a_gemm.cu:
      ```cpp
      if (num_tokens <= 8) {
          invokeFusedAGemm<__nv_bfloat16, kHdIn, kHdOut, 8>(out, mat_a, mat_b, ...);
      } else {
          invokeFusedAGemm<__nv_bfloat16, kHdIn, kHdOut, 16>(out, mat_a, mat_b, ...);
      }
      ```

  (4) PROGRAMMATIC DEPENDENT LAUNCH (PDL):
      Uses SM90+ PTX instructions to overlap this kernel's tail with the next kernel's launch,
      reducing inter-kernel latency in the CUDA graph pipeline.

      Source: csrc/dsv3_fused_a_gemm.cu:
      ```cpp
      asm volatile("griddepcontrol.wait;");            // Wait for prior kernel data
      asm volatile("griddepcontrol.launch_dependents;"); // Signal next kernel to start
      ```

  (5) RAW PTX FOR ALL CRITICAL PATHS:
      Uses inline PTX mma.sync.aligned.m16n8k16 for Tensor Core MMA, cp.async for async
      global->shared loads with L2 cache hints, and ldmatrix for efficient shared->register
      loads. Avoids all CUTLASS abstraction overhead.

  COMMIT REFERENCE:
    6874638bc — "[Model Bash] DeepSeek R1 BF16 Min Latency QKV A GEMM (0.5% E2E Speedup)"
    PR #34758, Author: Robert Shaw (robshaw@redhat.com), Feb 18, 2026
    855 insertions across 7 files. Introduced DeepSeekV2FusedQkvAProjLinear and dsv3_fused_a_gemm.

  Python dispatch (deepseek_v2.py, lines ~808-860):
  ```python

      def __init__(self, ...):
          super().__init__(...)
  ```

 A.2  MoE Gate Router GEMM  [Match C1]
      TARGET: 2,913 ns  |  Baseline: 10,880 ns  |  Delta: -7,967 ns (-73.2%)
      Category: MAJOR IMPROVEMENT — second largest single-op saving
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  The MoE gate/router projection GEMM [7168 -> 384] was replaced from a generic cuBLAS/nvjet
  CUTLASS BF16 GEMM to a purpose-built, one-block-per-expert CUDA kernel with PDL support.

  Baseline (v0.16.0):
    Kernel: nvjet_tst_8x64_64x16_4x1_v_bz_TNN  (generic cuBLASLt CUTLASS BF16 GEMM)
    Grid: (48,1,1)  |  Duration: 10,880 ns
    Dispatch path: ReplicatedLinear -> F.linear -> cuBLAS (no GateLinear class existed)

  TARGET (v0.20.2):
    Kernel: dsv3_router_gemm  (specialized router GEMM from TRT-LLM)
    Grid: (384,1,1)  |  Duration: 2,913 ns
    Dispatch path: GateLinear -> ops.dsv3_router_gemm

  ROOT CAUSE:
  The dsv3_router_gemm kernel (csrc/moe/dsv3_router_gemm_float_out.cu) uses a fundamentally
  different parallelization strategy than generic CUTLASS:

  (1) ONE-BLOCK-PER-EXPERT MAPPING:
      Grid = kNumExperts = 384. Each threadblock computes the full dot-product of all tokens
      (1-16) against one expert's weight row (7168-dim). This is a perfect decomposition for
      the extremely skinny GEMM shape [M<=16, N=384, K=7168].

      The old cuBLAS kernel used a tile-based decomposition with grid=(48,1,1), meaning each
      block computed an 8-column tile of the output. This causes poor load balance and
      unnecessary tile-management overhead for such a skinny problem.

      Source: csrc/moe/dsv3_router_gemm_bf16_out.cu:
      ```cpp
      __global__ __launch_bounds__(128, 1)
      void router_gemm_kernel_bf16_output(out, mat_a, mat_b) {
          int const n_idx = blockIdx.x;  // One block per expert
          // Each of 128 threads processes VPT=8 elements per K iteration
          // K=7168 / (128*8) = 7 iterations, fully unrolled at compile time
      }
      ```

  (2) COMPILE-TIME TEMPLATE SPECIALIZATION:
      kNumTokens (1-16), kNumExperts (256 or 384), and kHiddenDim (7168) are all compile-time
      template parameters. Every loop bound is known, enabling full unrolling and eliminating
      all runtime branching.

  (3) WARP-LEVEL SHUFFLE REDUCTION:
      The K-dimension reduction uses __shfl_xor_sync butterfly pattern (no shared memory),
      followed by a minimal 4-warp shared-memory cross-warp reduction.

  (4) PDL SUPPORT for kernel pipelining (same as dsv3_fused_a_gemm).

  GateLinear dispatch logic (gate_linear.py) implements a 3-tier strategy:
  ```python
  # Tier 1: dsv3_router_gemm — SM90+, batch<=16, dims 7168x{256,384}
  if self.allow_dsv3_router_gemm and x.shape[0] <= 16:
      output = ops.dsv3_router_gemm(x, self.weight, ...)
      return output, None

  # Tier 2: cuBLAS bf16->fp32 — SM90+, any batch, bf16 weights
  if self.allow_cublas_router_gemm and x.dtype == torch.bfloat16:
      output = ops.router_gemm_bf16_fp32(x, self.weight)
      return output, None

  # Tier 3: F.linear fallback
  ```

    8435b2e04 — "[ModelBash][DSV3] Add TRTLLM DSV3 Router GEMM kernel (6% B1 Speedup)"
    PR #34302, Author: Robert Shaw (robshaw@redhat.com), Feb 23, 2026

 A.3  MoE Routing Dispatch Consolidation  [Match C5 vs C5a-C5d]
      TARGET: 7,616 ns (1 kernel)  |  Baseline: 15,264 ns (4 kernels)  |  Delta: -7,648 ns (-50.1%)
      Category: MAJOR IMPROVEMENT — kernel fusion eliminates 3 launch overheads
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  The MoE routing pipeline — sigmoid scoring, bias correction, top-k expert selection, and
  token-to-expert index clustering — was consolidated from 4 separate PyTorch-dispatched
  CUDA kernels into a single monolithic TRTLLM kernel.

  Baseline (v0.16.0) — 4 separate kernels on main stream:
    moe_routing_sigmoid          2,400 ns  — vectorized_elementwise_kernel (sigmoid)
    moe_routing_bias_correction  3,200 ns  — unrolled_elementwise_kernel (add bias)
    moe_topk_routing_main        4,352 ns  — routingDeepSeek topk selection
    moe_routing_indices          5,312 ns  — routingDeepSeek index clustering
    Total:                      15,264 ns  (4 kernel launches)

  TARGET (v0.20.2) — 1 monolithic kernel:
    moe_routing_dispatch         7,616 ns  — routingIndicesBlockKernel (fused all steps)

  ROOT CAUSE:
  In v0.16.0, the MoE layer took the non-monolithic path, which called
  `GroupedTopKRouter._compute_routing()` in Python. This uses standard PyTorch ops compiled
  via torch.compile:

  Old path (grouped_topk_router.py):
  ```python
  scores = gating_output.sigmoid()                    # -> separate CUDA kernel
  scores = scores + e_score_correction_bias.unsqueeze(0)  # -> separate CUDA kernel
  group_scores = scores.view(...).topk(2, dim=-1)[0].sum(dim=-1)  # -> separate kernel
  topk_ids = torch.topk(tmp_scores, k=topk, ...)     # -> separate CUDA kernel
  ```

  In v0.20.2, the FusedMoE layer checks `self.quant_method.is_monolithic` and, when true,
  passes raw router logits directly to the TRTLLM kernel via FlashInfer:

  New path (layer.py, lines 1924-1929):
  ```python
  if self.quant_method.is_monolithic:
      final_hidden_states = self.quant_method.apply_monolithic(
          layer=self, x=x, router_logits=router_logits)
  ```

  The monolithic path calls FlashInfer's `trtllm_fp8_block_scale_moe()`, which passes
  `routing_logits` (raw, not pre-routed) and `routing_bias` to a single TRTLLM kernel:

  ```python
  flashinfer_trtllm_fp8_block_scale_moe(
      routing_logits=routing_logits,      # raw logits — NOT topk results
      routing_bias=routing_bias,           # e_score_correction_bias
      routing_method_type=RoutingMethodType.DeepSeekV3,  # enum = 2
      ...
  )
  ```

  Inside TRTLLM (flashinfer's pre-compiled cubin), the `routingMainKernel` performs all
  routing steps in shared memory within one kernel launch:
    - sigmoid activation on scores
    - bias correction (e_score_correction_bias addition)
    - group-level top-k via warp-level reductions
    - expert-level top-k selection
    - weight normalization and permuted index generation

  The monolithic path was enabled by the combination of:
    - FlashInfer upgrade from 0.6.3 to 0.6.8.post1 (improved TRTLLM MoE support for B200)
    - vLLM backend selection logic routing to Fp8MoeBackend.FLASHINFER_TRTLLM on SM100

  WHY THE MONOLITHIC PATH IS FASTER:
    - Eliminates 3 kernel launch overheads (~500-800 ns each on B200)
    - Data stays in shared memory/registers between routing stages (no global memory round-trips)
    - The TRTLLM kernel is hand-optimized for DeepSeek V3 routing (sigmoid + noaux_tc bias)

  NOTE: The monolithic path introduces 2 new small ops on the main stream:
    - moe_route_buffer_init (1,631 ns) — workspace buffer zeroing for routing arrays
    - moe_input_bf16_cast (1,665 ns) — dtype cast for TRTLLM input format
    These partially offset the savings (see Section B.1 below).

 A.4  Shared Expert Down Projection FP4 GEMM  [Match E7]
      TARGET: 5,343 ns  |  Baseline: 10,112 ns  |  Delta: -4,769 ns (-47.2%)
      Category: NOTABLE IMPROVEMENT — nearly 2x faster on aux stream
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  The shared expert down projection NVFP4 GEMM ([256/GPU -> 7168]) on the auxiliary stream
  became nearly 2x faster, despite using the same FlashInfer CUTLASS NvFp4 kernel type.

  Baseline (v0.16.0):
    Kernel: FlashInfer CUTLASS NvFp4 device_kernel
    Grid: (1,56,1)  |  Duration: 10,112 ns
    FlashInfer version: 0.6.3

  TARGET (v0.20.2):
    Kernel: FlashInfer CUTLASS NvFp4 device_kernel
    Grid: (2,56,1)  |  Duration: 5,343 ns
    FlashInfer version: 0.6.8.post1

  ROOT CAUSE:
  The grid dimension change from (1,56,1) to (2,56,1) reveals a CTA cluster configuration
  change in FlashInfer's CUTLASS FP4 GEMM between versions 0.6.3 and 0.6.8.post1.

  FlashInfer 0.6.8 switched from a ClusterShape of <1,1,1> to <2,1,1> for this problem size.
  On SM100 (B200), CUTLASS CTA clusters with <2,1,1> enable multi-SM cooperative execution
  via multicast, effectively doubling the M-dimension parallelism. This is especially
  beneficial for narrow-M problems (BS=1 decode).

  For reference, vLLM's own SM100 FP4 config follows the same pattern:

  Source: csrc/quantization/fp4/nvfp4_scaled_mm_kernels.cu:
  ```cpp
  struct sm100_fp4_config_default {
      using ClusterShape = Shape<_2, _1, _1>;    // 2x1x1 cluster for larger M
  };
  struct sm100_fp4_config_M16 {
      using ClusterShape = Shape<_1, _1, _1>;    // 1x1x1 cluster for M<=16
  };
  ```

  The FlashInfer 0.6.8 likely updated its heuristics to use the <2,1,1> cluster for the
  shared expert down-proj shape, matching the more aggressive SM100 tile configuration.

  The vLLM dispatch path is unchanged between versions:
  ```python
  # nvfp4_utils.py — both versions
  if current_platform.has_device_capability(100) and has_flashinfer():
      backend = NvFp4LinearBackend.FLASHINFER_CUTLASS
  ```

  IMPACT ON WALL TIME:
  This op runs on the auxiliary shared-expert stream (S5644), overlapping with the routed
  MoE on the main stream (S23). Its wall-time impact depends on whether the aux stream is
  the critical path. With the down GEMM now 5.3 us (vs 7.9 us for the concurrent
  moe_expert_gemm2_down on main stream), the aux stream is no longer a bottleneck for
  this segment.

 A.5  Post-Attention Allreduce + RMSNorm + Memcpy Elimination  [Match B12 + B13]
      TARGET: 8,447 ns  |  Baseline: 11,072 ns (10,016 + 1,056)  |  Delta: -2,625 ns (-23.7%)
      Category: MEANINGFUL IMPROVEMENT — protocol change + kernel elimination
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  The fused allreduce+residual_add+RMSNorm kernel after the attention O-projection became
  1,569 ns faster, AND a separate memcpy32_post Lamport signaling kernel (1,056 ns) was
  eliminated entirely.

  Baseline (v0.16.0):
    fused_allreduce_post_attn_rmsnorm  10,016 ns  — flashinfer trtllm_allreduce_fusion
    allreduce_completion_memcpy         1,056 ns  — memcpy32_post (Lamport barrier signal)
    Total:                             11,072 ns

  TARGET (v0.20.2):
    fused_allreduce_rms_post_attn       8,447 ns  — flashinfer allreduce_fusion (unified API)
    (no memcpy)
    Total:                              8,447 ns

  ROOT CAUSE:
  The FlashInfer allreduce API was migrated from the old `trtllm_allreduce_fusion` to a new
  unified `allreduce_fusion` API.

  Key commit: d4f123cc4 — "[Kernel] FlashInfer: switch allreduce fusion to unified API"
  PR #33985, Feb 9, 2026

  Old API (v0.16.0, allreduce_rms_fusion.py):
  ```python
  flashinfer_comm.trtllm_allreduce_fusion(
      ...,
      trigger_completion_at_end=True,   # <-- separate memcpy for Lamport signaling
      use_fp32_lamport=True/False,
      workspace_ptrs=_FI_WORKSPACE_TENSOR,
  )
  ```

  New API (v0.20.2, allreduce_rms_fusion.py):
  ```python
  flashinfer_comm.allreduce_fusion(
      ...,
      workspace=_FI_WORKSPACE,  # opaque workspace object
      # trigger_completion_at_end REMOVED — internalized in kernel
  )
  ```

  In the old protocol, `trigger_completion_at_end=True` caused a separate `memcpy32_post`
  kernel to be launched after the fused allreduce+RMSNorm kernel, writing a 32-byte signal
  to indicate that this rank's contribution was complete. Other ranks would read this signal
  before accessing the allreduced result.

  In the new unified API (FlashInfer 0.6.8.post1), the completion signaling is internalized
  into the main kernel's epilogue — no separate kernel launch is needed. This eliminates:
    - The 1,056 ns memcpy32_post kernel
    - The kernel launch overhead between the allreduce and the memcpy
    - The main kernel itself is ~1,569 ns faster (likely from simplified barrier logic)

 B.1  New TRTLLM MoE Buffer Init + BF16 Cast Ops  [Match C2 + C4]
      TARGET: +3,296 ns (NEW)  |  Baseline: not present
      Category: NEW OPS — overhead from TRTLLM monolithic MoE pipeline
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  Two new operations appeared on the main stream as part of the TRTLLM monolithic MoE kernel
  pipeline that did not exist in the baseline's non-monolithic routing path.

  moe_route_buffer_init    1,631 ns  (S23)  — memset for TRTLLM routing workspace buffers
  moe_input_bf16_cast      1,665 ns  (S23)  — BF16 copy/cast for TRTLLM input format

  ROOT CAUSE:
  These are pre-compiled TRTLLM cubin helper kernels (from FlashInfer's NVIDIA artifactory
  downloads). They exist because the TRTLLM monolithic MoE kernel must:

  (1) moe_route_buffer_init: Zero-initialize routing workspace arrays (expert count histograms,
      permuted index arrays, CTA mapping arrays) before the routingMainKernel writes to them
      atomically. These buffers are part of the TRTLLM MoE runner data structures.

  (2) moe_input_bf16_cast: Convert hidden states to the expected BF16 format for the TRTLLM
      routing kernel. The Python side also casts routing_logits to float32 for DeepSeek V3
      routing (flashinfer_trtllm_moe.py line 213).

  In the baseline's non-monolithic path, routing was performed via standard PyTorch ops
  (sigmoid, add, topk) that did not need separate buffer initialization — they used
  PyTorch's implicit memory management.

  OPTIMIZATION OPPORTUNITY:
  These buffer inits could potentially be eliminated by:
    - Pre-allocating persistent workspace buffers (avoid per-call zeroing)
    - Fusing the memset into the routingMainKernel's prologue
    - Using non-zeroing allocation if the kernel handles uninitialized values
  Estimated recovery: up to 3.3 us per layer.

 B.2  New Shared Expert Buffer Init Ops  [Match E1 + E5]
      TARGET: +2,751 ns (NEW)  |  Baseline: not present
      Category: NEW OPS — aux stream buffer allocation overhead
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  Two FillFunctor memset operations appeared on the auxiliary shared-expert stream in the
  TARGET, initializing output buffers for the CUTLASS FP4 GEMM calls.

  shared_expert_gateup_buf_init  1,600 ns  (S5644)  — output buffer for gate+up GEMM
  shared_expert_down_buf_init    1,151 ns  (S5644)  — output buffer for down GEMM

  ROOT CAUSE:
  In v0.20.2, the shared expert MLP runs on a dedicated auxiliary CUDA stream via the
  `SharedExperts` class (vllm/model_executor/layers/fused_moe/runner/shared_experts.py):

  ```python
  def _run_in_aux_stream(self, shared_experts_input):
      with torch.cuda.stream(self._stream):
          output = self._layer(shared_experts_input)
      current_stream().wait_stream(self._stream)
      return output
  ```

  When the CUTLASS FP4 GEMM allocates output tensors via `torch.empty()` on the aux stream,
  the CUDA allocator may need to initialize memory from a different pool than the main
  stream, producing visible `cudaMemsetAsync` calls that show up as `buf_init` kernels.

  In v0.16.0, shared experts also ran on an aux stream, but the buffer initialization was
  either amortized into the GEMM launch or used pre-warmed memory regions. The FlashInfer
  0.6.8 CUTLASS kernel likely requires explicit buffer initialization for its epilogue
  (matching the grid change to 2x cluster).

  Key commits for dual-stream shared expert execution:
    344a0017c — "[Performance] Dual stream execution of shared_experts and selected_experts"
    7cf56a59a — SharedExperts class refactor
    726efe177 — Additional shared expert runner changes

  IMPACT ON WALL TIME:
  These ops run on the aux stream, so their wall-time impact depends on whether they extend
  the aux stream's critical path beyond the main stream. Since the main stream's MoE routing
  + expert GEMMs take longer than the aux stream's shared expert pipeline (even with these
  inits), the wall-time impact is minimal.

 B.3  Post-MoE Allreduce Regression  [Match A2]
      TARGET: 9,024 ns  |  Baseline: 8,064 ns  |  Delta: +960 ns (+11.9%)
      Category: MINOR REGRESSION — allreduce variance or tuning difference
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  The fused EP/TP allreduce + residual_add + RMSNorm after the MoE layer (which also serves
  as the NEXT layer's input normalization via cross-layer fusion) is 960 ns slower.

  Both versions use the same flashinfer trtllm allreduce_fusion kernel with identical grid
  (4,1,1) and the same AllReduceFusedAddRMSNormPattern. The difference is the underlying
  FlashInfer API:
    - v0.16.0: flashinfer_comm.trtllm_allreduce_fusion()
    - v0.20.2: flashinfer_comm.allreduce_fusion() (unified API)

  ROOT CAUSE:
  The unified allreduce API in FlashInfer 0.6.8.post1 internalizes the completion signaling
  (which was previously a separate memcpy32_post kernel). For the post-attention allreduce,
  this internalization was a net win (-2,625 ns). For the post-MoE allreduce, it added a
  slight overhead (+960 ns), possibly because:
    - Different Lamport barrier tuning for the EP/TP allreduce pattern (EP+TP vs pure TP)
    - The cross-layer fusion pattern (which feeds the next layer's input) may have slightly
      different synchronization requirements in the unified API
    - Normal GPU execution variance (the block CoV for v0.20.2 is 0.95%)

  This is partially offset by the elimination of memcpy32_post at the post-MoE position
  (which would have cost ~1,056 ns in v0.16.0 but isn't counted here because it's at a
  different block boundary position).

 B.4  Q-B Projection GEMM  [Match B4]
      TARGET: 5,920 ns  |  Baseline: 5,280 ns  |  Delta: +640 ns (+12.1%)
      Category: MINOR REGRESSION — cuBLASLt version change
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  The Q-B expansion GEMM (ColumnParallel [1536 -> 1536/GPU]) regressed by 640 ns despite
  using the same nvjet CUTLASS kernel name and grid dimensions in both versions.

  Both versions:
    Kernel: nvjet_tst_16x64_64x16_4x1_v_bz_TNN
    Grid: (96,1,1)
    Dispatch: ColumnParallelLinear -> F.linear -> cuBLASLt

  ROOT CAUSE:
  The regression is caused by the CUDA toolkit upgrade from 12.9.1 to 13.0.2, which ships
  a different cuBLASLt version. The nvjet CUTLASS kernels are JIT-compiled by cuBLASLt at
  runtime, and CUDA 13.0.2's cuBLASLt may generate slightly different instruction sequences,
  scheduling strategies, or register allocation for the same tile configuration.

  Evidence:
    - v0.16.0: CUDA 12.9.1, PyTorch 2.9.1
    - v0.20.2: CUDA 13.0.2, PyTorch 2.11.0

  The kernel name and grid are identical because the tile decomposition heuristic selected the
  same tile shape (16x64 with 4x1 warp arrangement). However, the internal codegen quality
  differs between cuBLASLt versions. This is a known phenomenon — cuBLASLt performance can
  vary +-15% between CUDA toolkit releases for specific problem sizes.

  The vLLM dispatch path is identical in both versions:
  ```python
  # utils.py — unchanged between versions
  def default_unquantized_gemm(layer, x, weight, bias=None):
      return torch.nn.functional.linear(x, weight, bias)
  ```

 B.7  Q-A LayerNorm Reduction  [Match B3]
      TARGET: 2,433 ns  |  Baseline: 2,080 ns  |  Delta: +353 ns (+17.0%)
      Category: MINOR REGRESSION — RMSNorm implementation change + Triton version
----------------------------------------------------------------------------------------------------------------------------

  WHAT CHANGED:
  The reduction phase of RMSNorm on the compressed Q representation (dim=1536) regressed
  by 353 ns. This is a Triton-compiled kernel in both versions, but the underlying RMSNorm
  implementation changed.

  ROOT CAUSE — TWO FACTORS:

  (1) RMSNorm IMPLEMENTATION CHANGE:
      v0.16.0 used a hand-optimized C++ CUDA kernel for RMSNorm:
      ```python
      # v0.16.0 layernorm.py
      def forward_cuda(self, x, residual=None):
          return rms_norm()  # calls torch.ops._C.rms_norm — custom CUDA kernel
      ```

      v0.20.2 moved to a pure-PyTorch implementation in `ir.ops.layernorm`:
      ```python
      # v0.20.2 ir/ops/layernorm.py
      def rms_norm(x, weight, epsilon, variance_size=None):
          orig_dtype = x.dtype
          x = x.to(torch.float32)
          variance = x.pow(2).mean(dim=-1, keepdim=True)
          x = x * torch.rsqrt(variance + epsilon)
          x = x.to(weight.dtype) * weight
          return x.to(orig_dtype)
      ```

      This pure-PyTorch implementation is graph-traceable by torch.compile/Inductor, which
      compiles it into the `triton_red_fused_2` reduction kernel. While this enables better
      fusion opportunities in the compiled graph, the generated Triton kernel is less optimized
      than the hand-tuned C++ CUDA kernel for this specific dimension (1536).

  (2) TRITON VERSION UPGRADE:
      v0.16.0: Triton 3.5.x (bundled with PyTorch 2.9.1)
      v0.20.2: Triton 3.7.x (bundled with PyTorch 2.11.0)

      Triton codegen quality can vary between versions for specific kernel patterns. The
      reduction kernel for dim=1536 may have different tile sizes or scheduling in 3.7.x.

  OPTIMIZATION OPPORTUNITY:
  The architectural decision to use ir.ops (graph-traceable) over custom C++ kernels is
  intentional — it enables torch.compile to fuse the RMSNorm with surrounding ops. The
  +353 ns regression is the cost of this architectural flexibility. A possible mitigation
  is to register the custom C++ kernel as a torch.compile-friendly custom op, getting both
  graph traceability and kernel performance.

SECTION 5: ACTIONABLE OPTIMIZATION IDEAS FOR TARGET
================================================================================================================================

  Despite the TARGET being 21.3% faster overall, the following regressions represent
  recoverable performance:

  1. cuBLASLt nvjet GEMM regressions (B4 + B5 + B6): +1,439 ns
     Action: Investigate CUDA 13.0.2 cuBLASLt heuristic tuning. Try setting
     CUBLASLT_WORKSPACE_SIZE, or use matmulPreference API to pin tile configs that
     were faster in CUDA 12.9.1. Alternatively, apply min-latency kernel treatment
     (like dsv3_fused_a_gemm) to Q-B and O-proj GEMMs for BS<=16.

  2. TRTLLM MoE buffer init overhead (C2 + C4): +3,296 ns
     Action: Pre-allocate persistent workspace buffers in the TRTLLM MoE runner to
     avoid per-call zeroing. Alternatively, fuse the memset into the routing kernel's
     prologue. This requires FlashInfer-side changes.

  3. Shared expert aux-stream buffer inits (E1 + E5): +2,751 ns
     Action: Pre-allocate output buffers for the shared expert MLP on the aux stream
     at initialization time, and reuse them across layers. This avoids per-call
     allocation/memset overhead. These ops are on the aux stream, so their wall-time
     impact depends on the aux stream being the critical path.

  4. Q-A LayerNorm reduction regression (B3): +353 ns
     Action: Register the hand-tuned C++ RMSNorm CUDA kernel (torch.ops._C.rms_norm)
     as a torch.compile-friendly custom op. This would preserve graph traceability
     while recovering the custom kernel's performance.

  5. Post-MoE allreduce regression (A2): +960 ns
     Action: Profile the unified allreduce API's Lamport barrier tuning for the
     EP/TP allreduce pattern. The post-attention allreduce improved (-1,569 ns) with
     the same API, suggesting the post-MoE pattern has different characteristics
     that the new API handles suboptimally.

  Total recoverable regression: ~8.8 us (9.1% of TARGET block time)
