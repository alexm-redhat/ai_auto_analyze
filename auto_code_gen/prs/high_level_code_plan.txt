================================================================================
SECTION 1: ROOT CAUSE ANALYSIS
================================================================================

Performance Gap:
  vLLM block wall time:  252.1 us
  SGLang block wall time: 188.9 us
  Gap:                    63.2 us (1.33x slower)

Sparse MLA Attention Kernel (Stage M) accounts for 51.7 us (82%) of the gap:
  vLLM:  63.78 us  (sparse_attn_fwd_kernel — FlashMLA C kernel, FP8 KV)
  SGLang: ~16.4 us (CUTLASS FA3 Sm90 device_kernel + combine, BF16 KV)

Five compounding causes make vLLM's kernel 4.9x slower:

  1. KERNEL ARCHITECTURE: FlashMLA C kernel lacks SM90 warpgroup MMA pipelining,
     TMA descriptor optimization, and fused Q-split support that CUTLASS FA3
     provides. SGLang uses flash_attn_with_kvcache from sgl_kernel (FA3 on SM90).

  2. FP8 KV CACHE IS LARGER THAN BF16: vLLM's FP8 format uses 656 bytes/token
     (512B FP8 NoPE + 16B scales + 128B BF16 RoPE) vs SGLang's 576 bytes/token
     (all BF16). The mixed layout adds dequantization overhead inside the kernel.

  3. HEAD PADDING WASTE: vLLM pads from 16 to 64 heads (4x compute waste).
     FlashMLA C kernel only supports h_q = 64 or 128. FA3 handles arbitrary
     head counts natively (16 heads with no padding).

  4. EXTRA METADATA OVERHEAD: vLLM's get_mla_metadata scheduling and H2D copies
     add latency. FA3 uses get_scheduler_metadata which runs on GPU.

  5. ADDITIONAL KERNEL OVERHEAD: vLLM runs 4 kernels for attention (cat Q,
     convert topk index, D2D memcpy for head padding, flash_mla kernel).
     SGLang runs 3 kernels (prepare_varlen_num_blocks, FA3, combine).

SECTION 2: KEY FINDING — vLLM ALREADY HAS THE REQUIRED KERNEL
================================================================================

Critical discovery: vLLM ALREADY has FA3 with full MLA + paged KV support on SM90:

  - File: vllm/vllm_flash_attn/flash_attn_interface.py
  - Function: flash_attn_varlen_func(fa_version=3)
  - C++ backend: _vllm_fa3_C.fwd (compiled .so)
  - Supports: block_table (paged KV), q_v (MLA qv parameter), num_splits
              (split-KV), seqused_k, scheduler_metadata, SM90 (Hopper)

  The existing FlashAttnMLAImpl (non-sparse) already uses this exact kernel path
  for full-sequence MLA decode:

    Source: vllm/v1/attention/backends/mla/flashattn_mla.py:334-353
    Call:   flash_attn_varlen_func(q=q_pe, k=k_pe_cache, v=kv_c_cache,
                q_v=q_nope, block_table=..., seqused_k=..., fa_version=3, ...)

  For sparse attention, the ONLY difference is:
    - block_table = topk_indices (sparse token selection, page_size=1)
    - seqused_k = topk count (2048) instead of full seq_len
    - KV cache reshaped for page_size=1 instead of page_size=block_size

  NOTE: The CuTe DSL FA4 (vllm_flash_attn/cute/interface.py) does NOT support
  paged KV on SM90 (assert blocks it). But FA3 C++ backend DOES support it
  (confirmed in .deps/vllm-flash-attn-src/hopper/flash_api.cpp).

SECTION 3: CODE ANALYSIS — SGLang (FASTER FRAMEWORK)

SGLang's sparse MLA decode path (from NSA backend, _forward_fa3):

  Source: sglang/srt/layers/attention/nsa_backend.py:1602-1638

  def _forward_fa3(self, q_rope, kv_cache, v_head_dim, q_nope, page_table,
                   cache_seqlens, cu_seqlens_q, cu_seqlens_k, max_seqlen_q,
                   sm_scale, logit_cap, page_size):
      # Split KV cache into rope and nope parts
      k_rope_cache = kv_cache[:, :, v_head_dim:]        # (total, 64)
      c_kv_cache = kv_cache[:, :, :v_head_dim]           # (total, 512)
      # Reshape for page_size=1, h_k=1
      k_rope_cache = k_rope_cache.view(-1, page_size, 1, qk_rope_dim)
      c_kv_cache = c_kv_cache.view(-1, page_size, 1, v_head_dim)
      # Call FA3 with paged sparse KV
      o = flash_attn_with_kvcache(
          q=q_rope,              # (B, H, 64) - rope queries
          k_cache=k_rope_cache,  # (total, 1, 1, 64) - rope keys
          v_cache=c_kv_cache,    # (total, 1, 1, 512) - nope/value
          qv=q_nope,             # (B, H, 512) - nope queries
          page_table=page_table, # (B, topk=2048) - sparse indices
          cache_seqlens=cache_seqlens,
          softmax_scale=sm_scale,
          num_splits=self.num_splits, ...)

  Key properties:
    - page_size = 1 (each "page" is one token)
    - page_table = topk_indices (sparse token positions in cache)
    - KV cache dtype = bfloat16 (576 bytes/token)
    - No head padding (FA3 handles 16 heads natively)
    - Split-KV enabled (num_splits > 1 for efficiency)

  Median block kernel timings:
    Op 663: prepare_varlen_num_blocks_kernel  3.36 us
    Op 664: FA3 Sm90 device_kernel           10.53 us
    Op 665: FlashAttnFwdCombine device_kernel 2.53 us
    TOTAL:                                   16.42 us

SECTION 5: IMPLEMENTATION PLAN — STEP-BY-STEP CODE CHANGES
================================================================================

Overview: Create a new FA3SparseMLABackend that inherits the FA3 approach from
FlashAttnMLAImpl but adapts it for sparse attention using topk_indices.

Changes are ordered from lower-level (kernel/data) to higher-level (backend/config).

------------------------------------------------------------------------------
STEP 1: Create FA3 Sparse MLA Backend Class
------------------------------------------------------------------------------

New file: vllm/v1/attention/backends/mla/fa3_sparse_mla.py

Port from: vllm/v1/attention/backends/mla/flashattn_mla.py (vLLM non-sparse FA3)
Reference: sglang/srt/layers/attention/nsa_backend.py:1602-1638 (SGLang FA3 sparse)

What is UNCHANGED from FlashAttnMLAImpl:
  - FA3 kernel call via flash_attn_varlen_func(fa_version=3)
  - MLA q_v parameter usage (q=q_pe, q_v=q_nope)
  - KV cache splitting: kv_c (512-dim) and k_pe (64-dim)
  - BF16 KV cache format (576 bytes/token, no FP8 quantization)
  - No head padding (FA3 handles 16 heads natively)

What is CHANGED from FlashAttnMLAImpl:
  - Inherits from SparseMLAAttentionImpl instead of MLACommonImpl
  - forward_mqa receives topk_indices from the Indexer
  - block_table = topk_indices converted to global slot indices (page_size=1)
  - seqused_k = topk_count tensor (all values = 2048) instead of seq_lens
  - scheduler_metadata computed with page_size=1, max_seqlen_k=topk
  - KV cache reshaped for page_size=1: (total_slots, 1, 1, dim) instead of
    (num_blocks, block_size, dim)

What is ported from SGLang:
  - The page_size=1 approach for sparse token-level access
  - topk_indices as page_table pattern (from nsa_backend._forward_fa3)
  - BF16 KV cache dtype auto-selection for DeepSeek V3.2 on SM90

Class structure:

  class FA3SparseMLABackend(AttentionBackend):
      accept_output_buffer = True
      supported_kv_cache_dtypes = ["auto", "bfloat16"]

      @staticmethod
      def get_supported_kernel_block_sizes():
          return [MultipleOf(1)]  # Any block size works; page_size=1 for sparse

      @staticmethod
      def get_name():
          return "FA3_SPARSE_MLA"

      @classmethod
      def supports_compute_capability(cls, capability):
          return capability.major == 9  # SM90 (Hopper/H200)

      @classmethod
      def is_mla(cls):
          return True

      @classmethod
      def is_sparse(cls):
          return True

  class FA3SparseMLAMetadataBuilder(AttentionMetadataBuilder):
      # Builds FA3 scheduler_metadata with page_size=1
      # Pre-allocates topk count tensor (all 2048s)
      # Uses get_scheduler_metadata from vllm_flash_attn

  class FA3SparseMLAImpl(SparseMLAAttentionImpl):
      def __init__(self, ...):
          # Store kv_lora_rank=512, qk_rope_head_dim=64, topk=2048
          # No FP8 workspace needed (BF16 cache)
          # No head padding (FA3 native)

      def forward_mqa(self, q, kv_c_and_k_pe_cache, attn_metadata, layer):
          # 1. Unpack q tuple: q_nope (B,H,512), q_pe (B,H,64)
          # 2. Get topk_indices from indexer buffer
          # 3. Convert topk_indices to global slot indices
          #    (reuse existing triton_convert_req_index_to_global_index)
          # 4. Split KV cache into kv_c and k_pe parts
          # 5. Reshape for page_size=1:
          #    k_pe_cache: (total_slots, 1, 1, 64)
          #    kv_c_cache: (total_slots, 1, 1, 512)
          # 6. Call flash_attn_varlen_func(
          #        q=q_pe, k=k_pe_cache, v=kv_c_cache,
          #        q_v=q_nope,
          #        block_table=topk_global_indices,  # (B, topk)
          #        seqused_k=topk_tensor,            # all 2048
          #        max_seqlen_k=topk,                # 2048
          #        fa_version=3,
          #        scheduler_metadata=...,
          #        num_splits=..., ...)
          # 7. Return (attn_out, lse_or_none)

Key implementation details:

  a. topk_indices conversion: Reuse vLLM's existing
     triton_convert_req_index_to_global_index from sparse_utils.py.
     This converts per-request relative topk indices to global cache slot
     indices. Output shape: (num_tokens, topk=2048).

  b. page_size=1 reshaping: The KV cache is stored as
     (num_blocks, block_size, 576). For page_size=1, reshape to
     (num_blocks * block_size, 1, 576), then split:
       k_pe: (total, 1, 1, 64)
       kv_c: (total, 1, 1, 512)
     The topk_indices (global slot indices) directly index into this.

  c. scheduler_metadata: Use get_scheduler_metadata with:
       page_size=1, max_seqlen_k=topk, cache_seqlens=topk_tensor
     Pre-compute during metadata build for CUDA graph compatibility.

  d. num_splits: Use auto-detect (num_splits=0) or compute heuristic.
     SGLang uses self.num_splits which is set based on GPU SM count.

  e. CUDA graph support: Pre-allocate topk_tensor, scheduler_metadata
     buffers at init. Update in-place during graph replay.

STEP 3: Metadata Builder — Simplified Scheduling
------------------------------------------------------------------------------

Modified file: New class in fa3_sparse_mla.py

Port from: vllm/v1/attention/backends/mla/flashattn_mla.py (FlashAttnMLAMetadataBuilder)

  - FlashMLASchedMeta and get_mla_metadata calls

  - MIN_HEADS_FOR_BF16_PREFILL logic

Implementation:

  class FA3SparseMLAMetadataBuilder(AttentionMetadataBuilder):
      _cudagraph_support = AttentionCGSupport.UNIFORM_BATCH

      def __init__(self, kv_cache_spec, layer_names, vllm_config, device):
          self.topk = vllm_config.model_config.hf_config.index_topk  # 2048
          self.num_heads = model_config.get_num_attention_heads(parallel_config)
          self.page_size = 1  # Sparse page_size=1

          # Pre-allocate topk count tensor (constant for all requests)
          max_num_seqs = vllm_config.scheduler_config.max_num_seqs
          self.topk_seqlen = torch.full(
              (max_num_seqs,), self.topk, device=device, dtype=torch.int32)

          # Pre-allocate scheduler_metadata buffer for CUDA graphs
          self.scheduler_metadata_buf = None  # Lazy init

      def build(self, common_prefix_len, common_attn_metadata, fast_build):
          # Build scheduler_metadata with page_size=1
          scheduler_metadata = get_scheduler_metadata(
              batch_size=num_reqs,
              max_seqlen_q=max_query_len,
              max_seqlen_k=self.topk,
              num_heads_q=self.num_heads,
              num_heads_kv=1,
              headdim=64,           # qk_rope_head_dim
              headdim_v=512,        # kv_lora_rank
              cache_seqlens=self.topk_seqlen[:num_reqs],
              page_size=self.page_size,
              num_splits=0,  # auto
          )
          return FA3SparseMLAMetadata(
              scheduler_metadata=scheduler_metadata,
              block_table=common_attn_metadata.block_table_tensor,
              req_id_per_token=...,
              topk_seqlen=self.topk_seqlen[:num_reqs], ...)

STEP 5: Block Size Compatibility
------------------------------------------------------------------------------

Modified files:
  - Block allocator / KV cache allocation

What changes:
  - FA3SparseMLABackend reports get_supported_kernel_block_sizes() = [MultipleOf(1)]
    This means any block size is acceptable for the underlying allocation.
  - The actual KV cache uses the system's default block size (e.g., 64 or 128)
    for memory allocation and page management.
  - The sparse attention kernel uses page_size=1 internally (virtual view),
    independent of the physical block size.

What stays the same:
  - Physical block allocation (num_blocks, block_size, 576) — unchanged
  - Block table for the KV cache manager — unchanged
  - Slot mapping for KV cache writes — unchanged

This is a virtual-vs-physical separation:
  - Physical: blocks of block_size tokens for memory allocation
  - Virtual: page_size=1 for sparse attention kernel (each slot is a "page")
  - The topk_indices are global slot indices that index into the flat cache

SECTION 10: RISK ASSESSMENT AND MITIGATIONS
================================================================================

Risk 1: FA3 page_size=1 performance on SM90
  Impact: If FA3 with page_size=1 is slower than expected due to non-TMA
          path for small pages on SM90.
  Mitigation: Benchmark FA3 page_size=1 vs page_size=64/128 early.
              SGLang already achieves ~10.5 us with this exact path.
              If slow, consider gathering sparse tokens into contiguous
              pages (page_size=64) before calling FA3 (adds ~0.8 us gather).

Risk 2: get_scheduler_metadata incompatibility with page_size=1
  Impact: Scheduler metadata might not handle page_size=1 correctly.
  Mitigation: Test scheduler metadata generation in Validation Step 3 first.
              Fallback: use num_splits=1 (bypasses scheduler metadata).

Risk 3: CUDA graph compatibility
  Impact: Dynamic topk_indices may break CUDA graph replay.
  Mitigation: Pre-allocate max-size buffers, update in-place.
              Follow FlashMLASparseMetadataBuilder's existing pattern.
              The topk_indices buffer is already CUDA-graph compatible
              (allocated at max batch size, sliced at runtime).

Risk 4: Numerical differences (BF16 vs FP8 KV cache)
  Impact: Output quality may change with BF16 KV cache.
  Mitigation: BF16 is strictly better precision than FP8.
              SGLang already uses BF16 KV cache for DSA without issues.
              Run log confirms: "Setting KV cache dtype to bfloat16 for
              DeepSeek DSA on SM9 device."

Risk 5: Page_size=1 memory bandwidth
  Impact: page_size=1 means more page table lookups (2048 per token).
  Mitigation: FA3's non-TMA path handles this efficiently.
              SGLang's profile confirms ~10.5 us kernel time with this path.
