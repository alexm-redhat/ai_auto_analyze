PR INFO: FA3-Based Sparse MLA Attention Backend for DeepSeek-V3.2
================================================================================

Plan Step: #1 — Sparse Attention → CUTLASS FA3
Priority:  CRITICAL
Est. Savings: ~51 us per transformer block (82% of the vLLM-SGLang gap)

================================================================================
SUMMARY
================================================================================

This PR adds a new FA3-based sparse MLA attention backend (FA3_SPARSE_MLA)
that replaces the slower FlashMLA C kernel (sparse_attn_fwd_kernel) with
vLLM's existing FlashAttention-3 kernel on SM90 (Hopper/H200).

Key insight: vLLM already has FA3 with full MLA + paged KV support on SM90
(via _vllm_fa3_C.fwd). The existing FlashAttnMLAImpl uses it for non-sparse
MLA decode. This PR creates a sparse variant using page_size=1 paged KV
with topk_indices as the block_table for sparse token-level access.

PERFORMANCE
================================================================================

  Test: DeepSeek-V3.2, FP8, H200, TP=8, ISL=4, OSL=1024, Batch=1

  Before (FLASHMLA_SPARSE):
    - sparse_attn_fwd_kernel: 63.78 us
    - Total sparse attention: 68.14 us
    - KV cache: FP8 (656 bytes/token)
    - Head padding: 16→64 (4x waste)

  After (FA3_SPARSE_MLA):
    - FA3 device_kernel: ~10-15 us (estimated from SGLang trace)
    - Total sparse attention: ~17 us (estimated)
    - KV cache: BF16 (576 bytes/token) — 12% smaller
    - No head padding needed

  Estimated block wall time improvement: 252.1 us → ~201.7 us (1.25x speedup)

================================================================================
KEY TECHNICAL DETAILS
================================================================================

1. FA3 C++ API (hopper/flash_api.cpp) verified to support:
   - Paged KV on SM90 (non-TMA path for page_size=1)
   - MLA qv parameter (head_size<=64, head_size_v>=256, SM90 only)
   - Different V headdim (head_size=64, head_size_v=512)
   - Split-KV with non-TMA paged KV

2. page_size=1 trick: Each KV cache token is treated as a "page",
   and topk_indices (global slot indices) serve as the block_table.
   FA3 uses non-TMA load path for page_size=1 (page_size % kBlockN != 0).

3. BF16 KV cache: 576 bytes/token (512 nope + 64 rope, all BF16).
   Smaller than FP8's 656 bytes/token (512B FP8 + 16B scales + 128B BF16).
